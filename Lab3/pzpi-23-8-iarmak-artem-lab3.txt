Тема роботи: Програмна система для контролю роботи співробітників служби доставки їжі.


Мета роботи:
На цій лабораторній роботі потрібно розробити front-end частину програмної системи.


Хід роботи


1. Опис прийнятих інженерних рішень
При розробці клієнтської частини веб-застосунку було прийнято такі архітектурні та інженерні рішення:
* Компонентно-орієнтована архітектура (React.js + Vite): Застосунок побудовано як Single Page Application. Використання Vite як інструменту збірки забезпечує миттєве гаряче перезавантаження та швидку компіляцію. Інтерфейс розбито на перевикористовувані функціональні компоненти.
* Декларативне керування станом (React Hooks): Для збереження локального стану сторінок та реактивного оновлення інтерфейсу використано стандартні хуки useState та useEffect.
* Рольова модель доступу (Рішення на базі Маршрутизації): За допомогою react-router-dom реалізовано клієнтську маршрутизацію. Доступ до робочих просторів розділено на рівні інтерфейсу: для кур'єра відмальовується персональний робочий стіл, для адміністратора - панель керування сутностями системи.
* Централізований HTTP-клієнт (Axios Wrapper): Взаємодія з REST API реалізована через ізольований модуль axiosClient. Він автоматично підтягує токен авторизації користувача з localStorage та додає його до заголовків HTTP-запитів.
* Динамічна інтернаціоналізація (i18next): Замість текстових констант використано бібліотеку react-i18next. Конфігурація підтримує мовні пакети з можливістю безшовного перемикання мови без перезавантаження сторінок застосунку.


2. Опис взаємодії компонентів web-застосунку одне з одним та з іншими елементами системи
Взаємодія елементів системи побудована на базі архітектурного шаблону клієнт-сервер із чітким розмежуванням обов'язків: 
Внутрішня взаємодія компонентів (Frontend):
1. App.jsx: Ініціалізує маршрутизатор BrowserRouter та монтує глобальний загальний компонент Header.
2. Header.jsx: Постійно відстежує стан авторизації за наявністю токена в localStorage. Якщо користувач авторизований, компонент рендерить індивідуальні елементи, що адаптуються під роль, а також викликає екшени зміни мови в модулі i18n.js.
3. AdminDashboard.jsx та Dashboard.jsx: Виступають в ролі розумних компонентів, які самостійно виконують асинхронні запити до сервера, зберігають масиви даних та передають їх у презентаційні компоненти, такі як OrderCard, через механізм props.
Взаємодія з іншими елементами системи (Backend + DB):
* Протокол взаємодії: Клієнтська частина спілкується з бекендом виключно через асинхронні HTTP-запити.
* Сценарій авторизації: Компонент Login відправляє облікові дані користувача. Сервер перевіряє їх через MariaDB та повертає JWT-токен і роль. Фронтенд зберігає їх локально і змінює стан відображення інтерфейсу.
* Сценарій розподілу замовлень: Адміністратор у AdminDashboard обирає замовлення та кур'єра. Запит PATCH /api/orders/:id/assign оновлює зв'язки в БД. Після цього при зміні стану тумблера кур'єра у Dashboard.jsx спрацьовує ефект, який через запит GET /api/orders/my?courierId=... витягує оновлені дані з бази даних та рендерить картку активного замовлення на екрані конкретного кур'єра.
3. UML Діаграма прецедентів (Use Case Diagram)
Актори: Неавторизований користувач, Кур'єр, Адміністратор.
Прецеденти для всіх: Авторизація, Зміна мови інтерфейсу.
Прецеденти для Кур'єра: Зміна статусу, Перегляд особистих замовлень, Завершення доставки замовлення.
Прецеденти для Адміністратора: Реєстрація нового кур'єра, Видалення кур'єра, Створення замовлення, Призначення замовлення на кур'єра, Експорт даних, Створення бэкапу БД.

4. UML Діаграма компонент (Component Diagram)
Ця діаграма показує фізичні частини програми:
* Client-Side Bundle (React App: Header, Login, Dashboard, AdminDashboard).
* i18n Localization Module (Конфігурація перекладів всередині i18n.js).
* HTTP Client Module (Централізований клієнт axiosClient.js).
* REST API Controllers (Серверні контролери: AuthController, CourierController, OrderController).
* Database Access Layer (Конфіг підключення до MariaDB через драйвер mysql2).
  

Рисунок 2 - UML Діаграма компонент


5. Додаткові діаграми
Обрано Activity Diagram та State Diagram, оскільки вони найкраще демонструють логіку розподілу та виконання замовлень у системі.
А. UML Діаграма діяльності (Activity Diagram)
Процес: "Призначення та виконання доставки замовлення".
1. Адміністратор створює замовлення та обирає активного кур'єра зі списку.
2. Система оновлює зв'язок у БД і змінює статус замовлення на in_progress.
3. Кур'єр переходить у статус "Онлайн", і фронтенд автоматично тригерить функцію fetchOrders.
4. На екрані кур'єра рендериться картка замовлення з адресою.
5. Кур'єр доставляє замовлення клієнту та натискає "Завершити замовлення".
6. Сервер фіксує геокоординати доставки та змінює статус замовлення в БД на completed.
7. Кінець.
  

Рисунок 3 - UML Діаграма діяльності
Б. UML Діаграма станів (State Diagram)
Об'єкт: "Життєвий цикл замовлення (Order Status)".
* States: New — замовлення успішно створене адміном, але ще не має виконавця.
   * InProgress — замовлення призначене конкретному кур'єру (courier_id записано в базу) і виконується.
   * Completed — доставку завершено, зафіксовано фінальні координати кур'єра, замовлення збережено в історію.
  

Рисунок 4 - UML Діаграма станів
6. Висновки
Розроблена full-stack система доставки на React та Node.js довела, що попереднє UML-моделювання дозволяє безпомилково реалізувати складну синхронізацію статусів замовлень між адміном і кур'єром. Застосунок забезпечує чітке розділення ролей, безпечну роботу з MariaDB та повністю готовий до масштабування завдяки впровадженню автоматизованих бекапів.
