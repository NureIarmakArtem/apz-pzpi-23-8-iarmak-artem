3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування
Шаблони (патерни) проєктування - це перевірені часом рішення типових проблем, які виникають під час розробки програмного забезпечення. Це не готовий код, який можна просто скопіювати в проєкт, а скоріше опис підходу того, як розв'язати конкретну задачу в архітектурі системи. Базовим стандартом є класифікація GoF за якою 23 класичні патерни поділяються на такі галузі залежно від мети:
* твірні (Creational) - оптимізують процес створення об'єктів, роблячи систему незалежною від того, як саме ці об'єкти створюються та збираються (Factory Method, Abstract Factory, Builder, Prototype, Singleton);
* структурні (Structural) - визначають правильні зв'язки між класами та об'єктами (Adapter, Bridge, Composite, Decorator);
* поведінкові (Behavioral) - налагоджують ефективну комунікацію та розподіл обов'язків між об'єктами, визначаючи, як вони взаємодіють один з одним (Command, State, Observer, Strategy).


Шаблони (патерни) проєктування — це перевірені часом, стандартизовані архітектурні підходи до розв'язання поширених проблем проєктування об'єктно-орієнтованого програмного забезпечення. Вони не є готовими фрагментами коду, а слугують загальною концепцією вирішення певного класу задач. Базовим стандартом у цій галузі є класифікація «Банди чотирьох» (GoF), згідно з якою 23 класичні патерни поділяються на три групи: твірні         , структурні (визначають способи побудови складних систем) та поведінкові (регламентують алгоритми взаємодії). Головна область застосування патернів полягає у створенні гнучкої, масштабованої архітектури, яка легко піддається супроводу.
Область застосування патернів полягає у створенні архітектури, що високої надійності, гнучкості та тривалої підтримки.




3.2 Опис обраного патерна
* Назва патерна: State
* Тип шаблона (патерна): поведінковий
* Основне призначення: дозволяє об'єкту змінювати свою поведінку залежно від внутрішнього стану. З боку здається, що об'єкт змінює свій клас, оскільки на одні й ті самі методи він починає реагувати по-різному.
* Передумови застосування: патерн варто використовувати, коли поведінка об'єкта залежить від його стану і має змінюватися під час виконання; коли в коді накопичилося забагато складних умовних операторів (if-else або switch), які перевіряють значення полів об'єкта перед виконанням дії.
* Переваги: позбавляє від громіздких умовних конструкцій; дотримується принципу єдиної відповідальності, оскільки код кожного конкретного стану виноситься в окремий клас; спрощує додавання нових станів без зміни існуючих класів. 
* Недоліки: може призвести до надмірного збільшення кількості класів, якщо станів дуже багато або вони часто змінюються. 




        3.3 Структура шаблона (патерна)
* Опис структурних частин:
   * Контекст (Context): Об'єкт, поведінка якого змінюється під час виконання програми. Він зберігає посилання на об'єкт поточного стану та делегує йому виконання клієнтських запитів.
   * Інтерфейс стану (State): Спільний інтерфейс (або абстрактний клас), що визначає набір методів, які мають бути реалізовані кожним конкретним станом. Через цей інтерфейс Контекст взаємодіє з об'єктами станів.
   * Конкретні стани (Concrete States): Класи, що реалізують специфічну поведінку, притаманну певному стану об'єкта. Кожен клас містить логіку виконання дій та відповідає за переведення Контексту в наступний стан за певних умов.
   * Клієнт (Client): Код, який використовує Контекст для виконання бізнес-логіки. Клієнту не потрібно знати про внутрішню структуру станів або правила переходів між ними - він просто взаємодіє з інтерфейсом Контексту. 
* Опис взаємодії об’єктів: Клієнт ініціює запит через метод об'єкта Контекст. Контекст делегує виконання цього запиту поточному об'єкту Конкретного стану, з яким він пов'язаний у даний момент. Об'єкт стану виконує відповідну логіку і, за умови настання певних подій, ініціює зміну стану самого Контексту, підставляючи в нього новий екземпляр іншого класу стану. Таким чином, Контекст динамічно змінює свою поведінку для наступних звернень Клієнта, залишаючись для нього єдиною точкою входу. 
* UML-діаграми з поясненнями
  

Рисунок 3.1 - UML-діаграма класів State




На діаграмі зображено зв'язок реалізації (від Concrete State до State), зв'язок композиції/агрегації (від Context до State), а також зв'язок залежності (від Client до Context). Окремо виділено зворотний зв'язок залежності від Concrete State до Context, що відображає механізм ініціації зміни поточного стану. 




3.4 Реалізація шаблона (патерна)
* Опис реалізації мовою програмування C#: 
* Основні етапи: 
   * Створення базових інтерфейсів: розробка абстракції IMatchState, яка визначає контракт для керування життєвим циклом матчу через методи HandleStart та HandleEnd, а також створення класу MatchContext, який виступає єдиною точкою входу для клієнта та зберігає посилання на поточний стан.
   * Реалізація конкретних класів: створення класів WaitingState, InProgressState та FinishedState, кожен з яких реалізує інтерфейс IMatchState. Усередині кожного стану прописана власна логіка: перехід до наступного етапу через метод TransitionTo у контексті або блокування некоректних дій (виведення помилок логіки).
   * Демонстрація використання: у методі Main створюється об'єкт MatchContext з початковим станом очікування. Шляхом послідовних викликів методів Start() та End() демонструється, як об'єкт динамічно змінює свою поведінку та успішно обробляє або відхиляє запити залежно від поточної фази турніру.

1 // --- 1. Інтерфейс СТАНУ (State) ---
2 public interface IMatchState
3 {
4     void HandleStart(MatchContext context);
5     void HandleEnd(MatchContext context);
6 }
7
8 // --- 2. КОНКРЕТНІ СТАНИ (Concrete Products/States) ---
9 
10 // Стан: Очікування гравців
11 public class WaitingState : IMatchState
12 {
13     public void HandleStart(MatchContext context)
14     {
15         Console.WriteLine("[Waiting] Всі гравці готові. Починаємо матч...");
16         context.TransitionTo(new InProgressState());
17     }
18 
19     public void HandleEnd(MatchContext context)
20     {
21         Console.WriteLine("[Error] Неможливо завершити матч, що не розпочався.");
22     }
23 }
24
25 // Стан: Матч триває
26 public class InProgressState : IMatchState
27 {
28    public void HandleStart(MatchContext context)
29     {
30         Console.WriteLine("[Error] Матч уже запущений і триває.");
31     }
32 
33     public void HandleEnd(MatchContext context)
34     {
35         Console.WriteLine("[InProgress] Матч завершено. Система фіксує
36  результати...");
37         context.TransitionTo(new FinishedState());
38     }
39 }
40 
41 // Стан: Матч завершено
42 public class FinishedState : IMatchState
43 {
44    public void HandleStart(MatchContext context)
45     {
46         Console.WriteLine("[Error] Неможливо розпочати завершений матч. Створіть нове
47  лобі.");
48     }
49 
50     public void HandleEnd(MatchContext context)
51     {
52         Console.WriteLine("[Error] Матч уже знаходиться в архіві.");
53    }
54 }
55 
56 // --- 3. КОНТЕКСТ (Context/Creator) ---
57 public class MatchContext
58 {
59     private IMatchState _state;
60
61     public MatchContext(IMatchState initialState)
62     {
63         TransitionTo(initialState);
64     }
65
66     // Метод для зміни стану
67     public void TransitionTo(IMatchState state)
68     {
69         _state = state;
70         Console.WriteLine($"---> Система: Перехід до стану
71  {state.GetType().Name}.");
72     }
73 
74     // Клієнтські методи, що делегують роботу стану
75     public void Start() => _state.HandleStart(this);
76     public void End() => _state.HandleEnd(this);
77 }
78
79 // --- 4. КЛІЄНТСЬКИЙ КОД (Client) ---
80 class Program
81 {
82     static void Main(string[] args)
83     {
84         Console.OutputEncoding = System.Text.Encoding.UTF8;
85
86         // Створюємо матч у початковому стані очікування
87         var match = new MatchContext(new WaitingState());
88
89         // 1. Пробуємо завершити (буде помилка логіки стану)
90         match.End();
91
92         // 2. Стартуємо матч (перехід до InProgress)
93         match.Start();
94 
95         // 3. Пробуємо стартувати ще раз (помилка)
96         match.Start();
97 
98         // 4. Завершуємо матч (перехід до Finished)
99         match.End();
100
101         // 5. Спроба маніпуляцій після завершення
102         match.Start();
103
104         Console.ReadLine();
105     }
106 }
