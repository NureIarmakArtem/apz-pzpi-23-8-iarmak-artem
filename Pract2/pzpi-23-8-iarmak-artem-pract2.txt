3 ОПИС ВИКОНАНОЇ РОБОТИ
        3.1 Загальні відомості про систему
Назва системи: Twitch.
Розробник: Twitch Interactive, Inc.
Призначення системи: Провідна глобальна стрімінгова платформа для проведення прямих трансляцій. Основний фокус зосереджено на відеоіграх, кіберспортивних турнірах, музичному та творчому контенті. Цільова аудиторія налічує понад 31 мільйон щоденних активних користувачів.  
Основні функції: трансляція потокового відео з низькою затримкою, інтерактивна взаємодія глядачів через чат, багаторівнева система монетизації, зберігання та перегляд архівів трансляцій (VOD), створення коротких кліпів.




        3.2 Архітектурний стиль системи
Мікросервісна архітектура у поєднанні з хмарною інфраструктурою.
Історичний перехід від монолітної структури до мікросервісів був викликаний необхідністю обробки екстремальних пікових навантажень та забезпечення мінімальної затримки при передачі відео. Це дозволяє команді Twitch незалежно масштабувати критичні компоненти системи - такі як чат, систему транскодування відео та механізми рекомендацій - а також паралельно розробляти та впроваджувати нові функції без ризику відмови всієї платформи.








        3.3 Основні компоненти системи
Опис підсистем:
* Інтерфейс користувача: Клієнтські додатки для Web , мобільних платформ, ігрових консолей та Smart TV. Забезпечують інтерактивну взаємодію глядача з контентом, відображення низькозатримного відео та роботу чату в реальному часі через WebSockets.
* Серверна частина: Мережа точок присутності (PoP) та вузлів Ingest, що приймають вхідні відеопотоки від стрімерів за протоколом RTMP. Включає кластери транскодування, які в реальному часі перетворюють відео у різні роздільні здатності для адаптивного стрімінгу.
* Рівень бізнес-логіки: Мікросервіси, що відповідають за керування життєвим циклом трансляцій, систему модерації, алгоритми персоналізованих рекомендацій та білінгову систему для обробки підписок і віртуальної валюти.
* Рівень зберігання даних: Розподілена система сховищ, що включає Amazon S3 для зберігання архівів трансляцій та кліпів, а також реляційні (PostgreSQL) та NoSQL (DynamoDB, Redis) бази даних для зберігання метаданих користувачів, логів чату та соціальних зв'язків.
* Допоміжні сервіси: Система сповіщень (Push-notifications), сервіси внутрішньої аналітики для стрімерів, рекламний сервер (Ad Server) та відкритий API для інтеграції сторонніх розробників і чат-ботів.




3.4 Взаємодія компонентів
Взаємодія компонентів Twitch побудована на принципі чіткого розділення керуючого трафіку (Control Plane) та потокових даних (Data Plane). Клієнтські пристрої відправляють метазапити через балансувальник навантаження AWS ELB, який спрямовує їх до API Gateway. Шлюз забезпечує авторизацію та, використовуючи Service Discovery, маршрутизує запит до відповідного мікросервісу (наприклад, Chat або Billing). Окремий архітектурний потік реалізований для стрімерів: вхідний сигнал через вузли Ingest потрапляє до Transcoding Service, де відео в реальному часі конвертується у різні роздільні здатності та зберігається в Amazon S3 для архівації. 
  

Рисунок 1.1 – Діаграма компонентів архітектури Twitch 


На діаграмі відображено високорівневу структурну організацію системи. Рисунок ілюструє застосування патерна «Polyglot Persistence», де вибір бази даних залежить від специфіки сервісу: Redis використовується для швидкої обробки повідомлень чату в реальному часі, PostgreSQL - для транзакційних даних користувачів, а DynamoDB - для високошвидкісного доступу до метаданих активних трансляцій. Таке розділення дозволяє незалежно масштабувати кожен компонент системи залежно від типу навантаження.
  

Рисунок 1.2 – Діаграма послідовності взаємодії користувача з системою 


Діаграма ілюструє покроковий потік даних під час типового сеансу глядача. Процес розділений на дві ключові фази:
* Фаза ініціалізації (API): Глядач отримує необхідні метадані (назву стріму, статус та токени доступу) через шлюз API та мікросервіси, що взаємодіють із базами даних.
* Фаза стрімінгу (Медіа): Після отримання маніфесту трансляції, клієнтський пристрій ініціює пряме з'єднання з вузлом Twitch Edge (CDN) для завантаження відеосегментів за протоколом HLS.
Така модель забезпечує ізоляцію важкого медіа-трафіку від бізнес-логіки системи. Це дозволяє підтримувати низьку затримку (Low Latency) при мільйонах одночасних підключень, оскільки хмарна інфраструктура AWS обробляє лише легкі JSON-запити, тоді як основне мережеве навантаження припадає на спеціалізовану мережу доставки контенту. 




        3.5 Масштабованість та надійність
* Механізми балансування навантаження: Використання AWS Elastic Load Balancing (ALB) для розподілу HTTP/gRPC запитів між мікросервісами та Global Server Load Balancing (GSLB) для спрямування глядачів на найближчі вузли Edge CDN.
* Відмовостійкість: Застосування патерна Circuit Breaker для ізоляції збоїв мікросервісів; архітектурне розділення Control Plane (API) та Data Plane (Відео) для стабільності трансляцій навіть при критичних збоях бази даних.
* Кешування: Розподілене кешування в пам'яті за допомогою Redis для миттєвого доступу до метаданих стрімів та статусів користувачів; глобальне кешування відео-фрагментів на периферійних серверах Twitch Edge.
* Реплікація: Синхронна реплікація в кількох зонах доступності (Multi-AZ) для PostgreSQL/RDS та географічне дублювання архівів трансляцій (VOD) у сховищах Amazon S3.
* Хмарна інфраструктура: Еластичне масштабування обчислювальних ресурсів за допомогою Amazon EKS (Kubernetes) для контейнеризованих Go-сервісів та автоматичне виділення потужностей для транскодування відео в реальному часі.
        


        3.6 Приклади програмної реалізації
Запит до ШІ (Gemini 3.1 Pro): "Згенеруй приклад того, як реалізується архітектурне розділення Control Plane та Data Plane на прикладі мови Go, який би демонстрував, як легкий мікросервіс обробляє API-запит і повертає лише метадані та посилання на CDN-маніфест, уникаючи прямої передачі важкого відеопотоку".


1 package main
2
3 import (
4         "encoding/json"
5         "net/http"
6 )
7 
8 // API-обробник (Control Plane), що повертає лише метадані
9 func streamStatusHandler(w http.ResponseWriter, r *http.Request) {
10         channelID := r.URL.Query().Get("id")
11
12         // Імітація миттєвої перевірки статусу в Redis
13         isLive := channelID == "twitch_main" 
14
15         response := map[string]interface{}{
16                "channel_id": channelID,
17                "is_live":    isLive,
18        }
19
20        // Якщо стрім іде, повертаємо посилання на CDN, а НЕ саме відео
21        if isLive {
22                 response["hls_url"] = "https://edge-cdn.twitch.tv/hls/" +
23  channelID + "/master.m3u8"
24    }
25
26         w.Header().Set("Content-Type", "application/json")
27         json.NewEncoder(w).Encode(response)
28 }


Функція streamStatusHandler виступає в ролі мікросервісу, що обробляє початковий запит глядача до системи. Завдяки імітації звернення до швидкого кешу Redis (рядок 13), сервіс миттєво перевіряє статус трансляції без навантаження на основну базу даних. Замість того, щоб ініціювати передачу самого відеопотоку через API, у разі активного стріму клієнту повертається лише посилання hls_url на відео-маніфест (рядок 22). Таким чином, код технічно обґрунтовує принцип «розділення керуючого трафіку та потокових даних» (Control Plane vs Data Plane), який дозволяє платформі Twitch уникати перевантаження рівня бізнес-логіки важким медіа-трафіком, делегуючи його безпосередню доставку спеціалізованій мережі периферійних серверів (Edge CDN).
